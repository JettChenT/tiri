Pinned Virtual Cursors API

Proposal for compositor-rendered cursors that are owned by IPC clients, pinned to window-relative coordinates, animated on movement, and usable for click, scroll, and text input without stealing normal pointer focus.

Target repo: tiri Surface: niri-ipc Render owner: compositor Coordinate space: window logical px

Goal

The new feature should let an external controller create one or more visible cursors that behave like durable annotations over a specific window. A cursor is pinned to a window id and a point inside that window. When the window moves, scrolls through the layout, changes output, or is rendered in a preview-like overlay, the cursor follows the same window-relative point. When the client moves the cursor to a new point, tiri animates the visual cursor from the old point to the new point.

These cursors are not additional Wayland seats. They are compositor-owned visual and action handles. Input injection should reuse the existing computer-use semantics: target a window by id, send click or scroll at a window-relative coordinate, and restore focus afterward.

Design constraint: this should not mutate the real hardware pointer location just to show agent or automation state. The normal cursor remains a single seat cursor; pinned cursors are separate render elements with explicit action APIs.

API Shape

Use normal niri IPC requests for stateful cursor management. Keep one-shot interaction commands under Action so the CLI and config conversion path remain familiar.

Core Types

Cursor Identity and Target

pub struct VirtualCursor {
    pub cursor_id: String,
    pub target: VirtualCursorTarget,
    pub position: WindowPoint,
    pub appearance: VirtualCursorAppearance,
    pub animation: VirtualCursorAnimation,
    pub visible: bool,
    pub z_index: i32,
}

pub struct VirtualCursorTarget {
    pub window_id: u64,
    pub behavior_when_missing: MissingWindowBehavior,
}

pub struct WindowPoint {
    pub x: f64,
    pub y: f64,
}
Field Meaning
cursor_id Stable client-chosen id. Reusing an id updates the existing cursor only when the request opts into replacement.
window_id The mapped window id from Request::Windows. Coordinates are logical pixels inside this window.
position The pinned point. This remains window-relative even when the window moves or changes output.
z_index Ordering among pinned cursors. All pinned cursors render above normal windows and below modal compositor UI unless configured otherwise.

Appearance

The appearance API should deliberately resemble the normal cursor configuration, but it also needs color overrides because automation cursors must be distinguishable from the real pointer and from each other. 

pub struct VirtualCursorAppearance {
    pub source: VirtualCursorSource,
    pub size: Option<u16>,
    pub color: Option<CursorColor>,
    pub outline_color: Option<CursorColor>,
    pub opacity: Option<f32>,
    pub label: Option<String>,
}

pub enum VirtualCursorSource {
    Theme {
        xcursor_theme: Option<String>,
        icon: Option<String>,
    },
    Builtin {

    },
    Surface {
        path: String,
        hotspot_x: i32,
        hotspot_y: i32,
    },
}

pub enum BuiltinCursorShape {
    Arrow,
    Crosshair,

    Dot,
    Hand,
}

pub struct CursorColor {
    pub r: f32,
    pub g: f32,
    pub b: f32,
    pub a: f32,
}

Theme Cursor

Uses the same xcursor loading path as the normal cursor. If color is set, colorization is a compositor tint over the rendered cursor texture.

Builtin Cursor

Uses simple compositor-drawn geometry. This is the most predictable path for colored agent cursors because fill, outline, size, and opacity are native parameters.

Recommended default: Builtin { shape: Ring }, color = rgba(45, 212, 191, 0.95), and a dark outline. It is readable on top of application content and clearly different from the hardware cursor.

Movement Animation

Movement is a state transition on the cursor. The cursor remains pinned to the same window, but its window-relative point interpolates from the current point to the requested point. During animation, rendering samples the animated point each frame; actions target the latest logical destination unless a command explicitly asks to wait. 

pub struct VirtualCursorAnimation {
    pub duration_ms: u32,
    pub curve: VirtualCursorCurve,
    pub interrupt: AnimationInterruptMode,
}

pub enum VirtualCursorCurve {
    Linear,
    EaseOutCubic,
    EaseInOutCubic,
    Spring {
        damping_ratio: f64,
        stiffness: f64,
        epsilon: f64,
    },
}

pub enum AnimationInterruptMode {
    RetargetFromCurrentVisualPosition,
    RetargetFromLastDestination,
    JumpToNewDestination,
}

Move Action

pub struct VirtualCursorMove {
    pub cursor_id: String,
    pub position: WindowPoint,
    pub animation: Option<VirtualCursorAnimation>,
    pub wait: bool,
}

wait = true means the IPC reply is delayed until the animation reaches the destination or the cursor is destroyed. For regular CLI use, wait = false keeps the request cheap and lets the compositor continue rendering animation frames.

Interaction Actions

Interaction actions use the cursor's current target window and either the destination point or the current visual point. The default should be destination point, because a controller that calls move then click normally expects the click to land at the requested coordinate, even if the visual animation is still finishing. 

pub struct VirtualCursorClick {
    pub cursor_id: String,
    pub button: PointerButton,
    pub count: u8,
    pub at: CursorActionPoint,
}

pub struct VirtualCursorScroll {
    pub cursor_id: String,
    pub scroll_x: i32,
    pub scroll_y: i32,
    pub at: CursorActionPoint,
}

pub struct VirtualCursorTypeText {
    pub cursor_id: String,
    pub text: String,
}

pub enum CursorActionPoint {
    Destination,
    CurrentVisual,
    Override(WindowPoint),
}

pub enum PointerButton {
    Left,
    Middle,
    Right,
    Back,
    Forward,
    Code(u32),
}

Under the hood, click and scroll can reuse the existing computer-use helpers: resolve cursor_id to window_id and a WindowPoint, then route through the same path as CuaClick and CuaScroll. That gives us window-relative targeting and avoids changing compositor layout focus.

CLI Sketch

The CLI should keep the common path terse and leave advanced styling to JSON. The names below map directly to the IPC action/request names. 

niri msg create-virtual-cursor \
  --cursor-id agent-primary \
  --window-id 42 \
  --x 320 --y 180 \
  --shape ring \
  --color '#8b5cf6' \
  --outline-color '#111827' \
  --size 28

niri msg action virtual-cursor-move \
  --cursor-id agent-primary \
  --x 580 --y 240 \
  --duration-ms 220

niri msg action virtual-cursor-click \
  --cursor-id agent-primary \
  --button left

niri msg destroy-virtual-cursor --cursor-id agent-primary

Behavior Rules

Case Proposed behavior
Window moves Cursor stays pinned to the same window-relative point and renders at the new global position.
Window resizes Point remains unchanged. If it falls outside the new logical bounds, render at the nearest visible edge but preserve the stored coordinate unless the client updates it.
Window closes Apply MissingWindowBehavior: hide, destroy, or keep stale state for reconnect/debugging.
Window is off-screen or on another workspace Cursor remains alive but does not render unless that window is currently rendered by the layout or a compositor UI that opts into pinned cursor overlays.
Normal hardware cursor moves No effect on pinned cursors.
Pinned cursor clicks Click targets the pinned window and coordinate using computer-use dispatch, without focusing the window unless requested by a future flag.
Screencopy Expose a separate policy flag later. Initial behavior should include pinned cursors in normal screen capture because they are visible compositor overlays. 
pub enum MissingWindowBehavior {
    Hide,
    Destroy,
    KeepStale,
}

Implementation Mapping

State

Add a VirtualCursorUi or PinnedCursorManager owned by Niri. Store cursors by id, with target window id, desired point, animated visual point, appearance, and render cache.

Rendering

Render pinned cursors in the compositor pass after windows and before high-priority UI. Resolve each cursor by asking whether its target mapped window is currently rendered and converting the window-relative point to output-relative physical coordinates.

Animation

Reuse the existing animation clock pattern. Any active cursor animation should make are_animations_ongoing return true and trigger redraws until complete.

Input

Route click, scroll, and text through the existing CUA helpers after resolving the cursor to window_id plus WindowPoint. This keeps interaction behavior independent from the real seat pointer.

Open Questions

  • Should pinned cursors appear inside window preview UI, MRU thumbnails, and screenshot window captures, or only on the normal desktop render path?
  • Should a cursor be allowed to pin to a layer-shell surface, or should step one only support mapped windows?
  • Should click actions ever focus the target window, or should focus-changing be a separate explicit action?
  • Should cursor colors support theme-aware names in addition to explicit RGBA values?
  • Should clients be able to subscribe to cursor lifecycle events, or is request-response listing enough for now?
Step one recommendation: implement mapped-window cursors only, include built-in colored shapes first, then add xcursor-theme source support after the render and animation model is stable.
